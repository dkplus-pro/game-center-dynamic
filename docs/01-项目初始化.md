## 为什么选择 Turborepo Monorepo

> Monorepo（单一仓库）是一种将多个项目/包放在同一个 Git 仓库中管理的策略。

### Monorepo 的优势

| 优势 | 说明 |
|------|------|
| **代码共享** | 公共类型定义、工具函数、组件只需维护一份，所有子项目引用同一份代码 |
| **统一工具链** | ESLint、TypeScript、Prettier 等配置集中管理，避免各子项目配置不一致 |
| **原子化变更** | 一次 commit 可以同时修改前端、后端、共享包，保证变更的完整性 |
| **简化 CI/CD** | 一条流水线即可构建、测试所有项目，无需管理多个仓库的 CI |

### 为什么选 Turborepo？

Turborepo 是 Vercel 推出的高性能构建系统，专为 JavaScript/TypeScript monorepo 设计：

- **极快的构建速度**：智能缓存机制，只重新构建变更的部分
- **并行执行**：自动分析依赖关系，最大化并行构建能力
- **远程缓存**（可选）：团队共享构建缓存，CI 环境也能享受缓存加速
- **零配置上手**：相比 Nx、Lerna 等工具，Turborepo 的配置更简洁直观

### 为什么用 pnpm？

- **严格的依赖管理**：pnpm 使用符号链接（symlink）管理 `node_modules`，避免幽灵依赖
- **节省磁盘空间**：全局存储 + 硬链接，相同版本的包只存一份
- **原生 workspace 支持**：`pnpm-workspace.yaml` 简单声明工作空间，无需额外工具

### 分层设计理念

```
┌─────────────────────────────────────┐
│           apps/admin                │  ← 管理后台应用
├─────────────────────────────────────┤
│           apps/h5                   │  ← H5 移动端应用
├─────────────────────────────────────┤
│   packages/components               │  ← 共享 UI 组件
├─────────────────────────────────────┤
│   packages/types                    │  ← 共享类型定义
├─────────────────────────────────────┤
│   packages/eslint-config            │  ← 代码规范配置
├─────────────────────────────────────┤
│   server/                           │  ← 后端服务
└─────────────────────────────────────┘
```

- **上层依赖下层**：app 可以依赖所有 package
- **下层不依赖上层**：package 不依赖任何 app
- **package 之间可以互相依赖**：例如 components 可以依赖 types

## pnpm Workspace 配置

### `pnpm-workspace.yaml`

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

这段配置告诉 pnpm：`apps/` 和 `packages/` 目录下的每一个子目录都是一个独立的包（workspace package）。

### `.npmrc`

```ini
shamefully-hoist=true
strict-peer-dependencies=false
```

| 配置项 | 作用 |
|--------|------|
| `shamefully-hoist=true` | 将依赖"提升"到根 `node_modules`，解决某些工具（如 ESLint）找不到插件的问题 |
| `strict-peer-dependencies=false` | 放宽 peerDependencies 校验，避免版本不匹配导致的安装失败 |

### 包命名规范

内部包使用 `@game-center/` 作用域前缀：

- `@game-center/eslint-config` — ESLint 共享配置
- `@game-center/types` — TypeScript 类型定义
- `@game-center/components` — 共享 UI 组件（Wave 2）

应用包使用项目名作为作用域：
- `@game-center/admin` — 管理后台（Wave 4）
- `@game-center/h5` — H5 端（Wave 3）

### 本地依赖引用

在 workspace 内部，使用 `workspace:*` 协议引用本地包（在 Wave 2/3/4 创建应用时会用到）：

```json
{
  "dependencies": {
    "@game-center/types": "workspace:*",
    "@game-center/components": "workspace:*"
  }
}
```

这样 pnpm 会自动链接到本地包，修改共享包后无需发布即可立即在所有应用中生效。

## 共享 ESLint 配置

### 为什么需要共享 ESLint 配置？

在 monorepo 中，如果每个子项目都维护自己的 ESLint 配置：

- 各项目的规则可能不一致，导致代码风格混乱
- 更新规则需要修改多个地方，维护成本高
- 新项目需要重新配置，增加创建成本

**解决方案**：将 ESLint 配置提取为一个独立的 workspace 包，所有子项目共同引用。

### 包结构

```
packages/eslint-config/
├── package.json    # 声明依赖（ESLint 插件）
└── index.js        # ESLint 配置规则
```

### 配置规则解析

```javascript
module.exports = {
  root: true,
  parser: '@typescript-eslint/parser',        // 使用 TypeScript 解析器
  parserOptions: {
    ecmaVersion: 'latest',                     // 最新 ECMAScript 语法
    sourceType: 'module',                      // ES Module
    ecmaFeatures: { jsx: true },               // 支持 JSX
  },
  plugins: ['@typescript-eslint', 'react', 'react-hooks'],
  extends: [
    'eslint:recommended',                      // ESLint 内置推荐规则
    'plugin:@typescript-eslint/recommended',   // TypeScript 推荐规则
    'plugin:react/recommended',                // React 推荐规则
    'plugin:react-hooks/recommended',          // React Hooks 规则
    'prettier',                                // 关闭与 Prettier 冲突的规则
  ],
  rules: {
    'react/react-in-jsx-scope': 'off',         // React 17+ 不需要 import React
    '@typescript-eslint/no-unused-vars': ['warn', { argsIgnorePattern: '^_' }],
    '@typescript-eslint/explicit-function-return-type': 'off',
  },
  ignorePatterns: ['dist', '.next', 'node_modules', '*.config.*'],
};
```

**关键点**：
- `eslint-config-prettier` 放在 extends 最后，确保 Prettier 规则覆盖 ESLint 规则，避免冲突
- 以 `_` 开头的未使用参数不会触发警告（如 `(_req, res)`）
- `react/react-in-jsx-scope` 关闭是因为 React 17+ 的新 JSX 转换不再需要手动 import

## Turbo 流水线配置

### `turbo.json` 任务定义

```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],           // 先构建所有依赖
      "outputs": ["dist/**", ".next/**"]  // 缓存输出目录
    },
    "dev": {
      "cache": false,                    // 开发模式不缓存
      "persistent": true                 // 持续运行（watch 模式）
    },
    "test": {
      "dependsOn": ["build"],            // 先构建再测试
      "outputs": []
    },
    "lint": {
      "dependsOn": ["^build"]            // 依赖包需先构建
    },
    "typecheck": {
      "dependsOn": ["^build"]            // 依赖包需先构建
    },
    "e2e": {
      "dependsOn": ["build"],
      "cache": false                     // E2E 测试不缓存
    },
    "clean": {
      "cache": false                     // 清理不缓存
    }
  }
}
```

### `dependsOn` 依赖拓扑说明

- `dependsOn: ["^build"]` — 插入符 `^` 表示"先构建此包的**依赖包**"
- `dependsOn: ["build"]` — 没有 `^`，表示"先执行此包自身的 build"

示例：假设 `apps/h5` 依赖 `packages/components` 和 `packages/types`：

```
运行 turbo build：

1. packages/types:build     ← 无依赖，直接构建
2. packages/components:build ← 依赖 types，等 types 构建完
3. apps/h5:build             ← 依赖 components + types，等两者都构建完

步骤 1-2 和 1-3 可以并行执行
```

### 根 package.json 脚本

```json
{
  "scripts": {
    "dev": "turbo dev",         // 启动所有 dev 任务
    "build": "turbo build",     // 构建所有包
    "test": "turbo test",       // 运行所有测试
    "lint": "turbo lint",       // 代码检查
    "typecheck": "turbo typecheck", // 类型检查
    "e2e": "turbo e2e",         // 端到端测试
    "clean": "turbo clean && rm -rf node_modules",
    "format": "prettier --check .",
    "format:fix": "prettier --write ."
  }
}
```

## Prettier + Husky 代码规范

### Prettier 配置

```json
{
  "semi": true,              // 语句末尾加分号
  "singleQuote": true,       // 使用单引号
  "trailingComma": "all",    // 所有可能的地方加尾逗号
  "printWidth": 100,         // 每行最多 100 字符
  "tabWidth": 2              // 缩进 2 空格
}
```

### Husky + lint-staged

Husky 用于管理 Git hooks，lint-staged 在提交前对暂存区文件运行检查和格式化：

```json
{
  "lint-staged": {
    "*.{ts,tsx,js,jsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md,yaml,css}": ["prettier --write"]
  }
}
```

根 `package.json` 中的 `"prepare": "husky"` 确保 `pnpm install` 后自动初始化 Git hooks。

> **注意**：完整的 Husky 初始化还需要运行 `npx husky init` 创建 `.husky/` 目录和 pre-commit hook，这将在本章的"开始开发"步骤中完成。

## 开始开发

### 前置条件

确保已安装以下工具：

- **Node.js** >= 18
- **pnpm** >= 9.15.0（推荐使用 `corepack enable && corepack prepare pnpm@9.15.0 --activate`）

### 初始化步骤

```bash
# 1. 克隆项目
git clone <repo-url> game-center-dynamic
cd game-center-dynamic

# 2. 安装依赖
pnpm install

# 3. 初始化 Husky
npx husky init
echo "pnpm lint-staged" > .husky/pre-commit

# 4. 验证配置
pnpm format           # 检查代码格式
pnpm format:fix       # 自动修复格式问题

# 5. 测试 turbo 流水线
pnpm build            # 构建所有包
pnpm lint             # 运行 lint
pnpm typecheck        # 运行类型检查
```

### 常用命令速查

| 命令 | 说明 |
|------|------|
| `pnpm install` | 安装所有依赖 |
| `pnpm dev` | 启动所有开发服务 |
| `pnpm build` | 构建所有包 |
| `pnpm test` | 运行所有测试 |
| `pnpm lint` | 代码规范检查 |
| `pnpm typecheck` | TypeScript 类型检查 |
| `pnpm e2e` | 端到端测试 |
| `pnpm clean` | 清理构建产物和 node_modules |
| `pnpm format` | 检查代码格式 |
| `pnpm format:fix` | 自动修复格式 |

### 常见问题

**Q: `pnpm install` 报错 `Unsupported engine`？**

确保 Node.js 版本 >= 18。使用 `node -v` 查看当前版本。

**Q: 修改了 `packages/eslint-config/index.js`，为什么 lint 没变化？**

Turborepo 有缓存机制，ESLint 配置变更后需要清除缓存：
```bash
pnpm clean && pnpm install && pnpm lint
```

**Q: 如何只运行某个子包的脚本？**

```bash
# 为特定包运行脚本
pnpm --filter @game-center/types build

# 为多个包运行
pnpm --filter @game-center/types --filter @game-center/components build
```
